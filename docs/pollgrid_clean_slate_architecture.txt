Vision & hard constraints

The pitch: take the actual engineering that makes GlobeGrid good — thread-per-source
ingestion that degrades instead of crashing, a permanent hash-chained fact chain,
correlation across sources and across decades, a bounded LLM that narrates but never
calculates, a hand-rolled WebGL2 renderer with real LOD, a buildless frontend with real
sound design and real theming — and point all of it at US elections instead of world events.
Not a redesign of the old PollGrid spec. A new one, built the way GlobeGrid was actually
built.


§ Non-negotiables

1. Primary sources only. Individual pollster releases, the FEC API, Census, the AP
   Elections API, state Secretary of State feeds, Congress.gov, official campaign transcripts
   and press releases. Never scrape RealClearPolitics / 538 / 270ToWin's aggregated
   numbers, ratings, or prose — link out to them, build every average from the underlying
   releases ourselves.

2. Race calls are never automated. The election-night engine flags a race CALLABLE for
   human review and stops there. election_night.auto_publish_calls is hardcoded
    False in code, not just config, and race_calls.called_by must be a real human
   identifier — never "system" , never a model name.

3. Append-only, everywhere. Poll averages, fundamentals snapshots, forecasts, and the
   volatility index are all dated snapshots on an as_of axis, never overwritten in place —
   the same mechanism that runs the archive back to 1950 also runs the time-capsule
   scrubber for a race that happened yesterday.

4. Provenance is hash-chained, not just logged. Every extracted fact and every prediction
   chains to the previous row's hash. It proves our own copy wasn't altered after being
   written — it does not, and never claims to, prove the underlying poll or statement was
   true.

5. The LLM is bounded and provider-agnostic. Local-first (Ollama), cloud fallbacks behind
   it. It writes race narratives, dossier prose, and rhetoric classifications inside a fixed rubric
   — it never touches poll averaging, the fundamentals model, forecasting math, correlation
   thresholds, the volatility index, or a race call. Every generative feature ships a
   deterministic fallback the pipeline actually exercises on failure.

6. Every tunable lives in one config file. Thresholds, intervals, weights, model names —
    config.yaml + .env , schema-validated at boot, fails the whole process loudly on a bad
   value rather than limping forward.

7. A failing source degrades, never crashes. Thread-per-source, exponential backoff
   capped at a config ceiling, three-state health ( ok / degraded / down ) — one dead
   pollster feed can't take down election night.

8. Synthetic data is always tagged and always purgeable in one operation.
    is_synthetic=1 , one command removes every trace.

The geopolitics-knowledge-layer trick, aimed at candidates instead of countries — and
pushed one tier deeper than GlobeGrid ever needed to go.

§ The three-tier acquisition pattern
Directly lifted from GlobeGrid's country/leader/party layer, because it's the single best idea in
that codebase — now doing double duty for candidates and for the demographic panel
underneath them:

1. Curated floor. Every sitting member of Congress, every current governor, and every
   major-party presidential/Senate/gubernatorial candidate gets a hand-seeded row with
   cited sources — so a candidate page is never blank on day one.

2. Free structured-API sync. FEC (finance), Census (demographics at every tier),
   Congress.gov (votes, bill sponsorship) refresh on a schedule, purely additive —
      COALESCE -style, never overwriting a curated value, and once a sync has touched a row
   the seed step leaves it alone.

3. LLM fills the remaining gaps — bio prose, a plain-language policy-position summary, a
   party-history paragraph — generated in the background, cached, merged over the floor,
   never blocking a page from opening.

Coverage isn't limited to the majors. Every candidate who has ever appeared on a
general-election ballot for President, Senate, Governor, or House gets a real page — a stub
the instant an official filing record or historical-archive import creates the row, thickening
automatically as sync and AI-fill reach it. A losing 1962 House candidate and a sitting
Senator both get a page; they just don't get the same depth on day one, and the page says
so.

Ingestion layer
One daemon thread per source, three-state health, a budget counter for anything rate-
capped.



The scheduler spawns one daemon thread per active source and re-reads that source's own
database row every tick, so toggling a source off or editing its poll interval takes effect on the
next cycle with no restart. Each thread waits on a shared stop-event rather than a plain
sleep, so shutdown is immediate. Backoff is exponential, capped at a configured ceiling,
same as the ground rules state.

§ Census, concretely
ingestion/census.py hits the ACS 5-year endpoint —
https://api.census.gov/data/{year}/acs/acs5?get=NAME,{variables}&for=
{geography}&in={parent}&key={CENSUS_API_KEY} — the 5-year product, not the 1-year,
because 1-year estimates only publish for geographies with 65,000+ population and this
platform needs county- and district-level reliability everywhere. Geography clauses differ by
tier exactly as the Census API requires: for=state:* , for=county:*&in=state:XX , and
— confirmed against the real API — for=congressional%20district:*&in=state:XX for
districts. Tract/block-group pulls ( for=tract:*&in=state:XX%20county:YYY ) are the raw
input to the precinct-derivation step below, never queried directly by a user-facing feature.

  THE REAL VINTAGE-MISMATCH PROBLEM — AND HOW IT'S HANDLED


  ACS congressional-district data is published against whichever Congress's boundaries were seated during the survey years — when a
  state redraws districts mid-decade, an ACS release can reflect the prior map for a year or more before the Bureau catches up. The fix
  isn't to ignore this: every Census pull is tagged with the specific Congress vintage it was published against, matched against the
   congressional_districts table's own effective_from / effective_to boundary versions — a demographic row is only ever joined
  to the district-shape version it actually describes, never silently mapped onto "whatever CD lines are current now."


Cadence: ACS 5-year estimates refresh annually (a rolling window); the decennial count
( dec/pl for the PL 94-171 redistricting file, dec/dhc for detailed demographic/housing
characteristics) lands once every ten years plus the mid-decade redistricting-file release.
Boundary geometry itself comes from TIGER/Line shapefiles (already the offline build
pipeline's source in The map) — TIGERweb, the live map-service version of the same data,
isn't needed since nothing here calls a live GIS service at runtime.

Precinct-level derivation, named precisely: population-weighted areal interpolation —
each Census block's population (from the decennial block-level count, the finest population
geography that exists) is distributed into whichever precinct(s) its geometry overlaps,
weighted by the fraction of the block's population estimated to fall in each; a precinct's
demographic estimate is then the population-weighted sum of the blocks assigned to it. This
is a real, named technique with well-understood error characteristics — it degrades when a
precinct boundary splits a Census block unevenly (rare, since precincts are usually drawn

§ FEC, concretely
ingestion/fec.py uses the real, free, public OpenFEC API ( api.open.fec.gov ) —
DEMO_KEY works for development at a throttled rate; a registered key raises the ceiling to
1,000 calls/hour. Endpoints: /candidates/ and /candidate/{candidate_id}/ for the
roster that seeds every historical and current candidate stub, /committees/ for
PAC/campaign-committee linkage, /schedules/schedule_a/ for itemized individual
contributions, /schedules/schedule_a/by_contributor/ for aggregated donor-side
rollups, and /schedules/schedule_b/ for itemized spending/disbursements — the
deterministic input to both the fundamentals model's fundraising-ratio term and the
corroboration engine's finance signal.


§ Congress.gov, concretely
ingestion/congress_gov.py uses the real, free (API-key-gated) Congress.gov API for
bills, sponsorship, and roll-call votes. Honest caveat: its House Roll Call Votes endpoint
currently only covers the 118th Congress forward (2023–present) — it is not a full historical
roll-call archive. For ideology scoring and roll-call history before 2023, the platform pulls from
VoteView (Poole & Rosenthal's DW-NOMINATE project, public-domain downloads covering
every Congress back to the 1st) — a real, load-bearing academic source rather than a gap
papered over, and one that hands the ideology-scoring model an actual peer-reviewed
methodology to sit on top of instead of inventing its own from scratch.

Extraction & the fact chain
Polls parse directly. Text gets NLP. Everything gets hash-chained before it's ever stored.



Polls need no NLP at all — they parse straight into a structured polls fact. News,
speeches, and transcripts go through: entity recognition (spaCy if installed, a deterministic
capitalized-run regex fallback otherwise — the same auto-detect-with-fallback idiom as every
other optional dependency in this design); a politics-tuned category classifier ( polling /
finance / legislation / endorsement / scandal / debate / election-result /
rhetoric / campaign-event , word-boundary keyword matching with a fixed tie-break
order so "war on drugs" doesn't trip a conflict keyword); entity canonicalization (candidate
and party name variants resolved via exact-alias cache, then fuzzy match, so "Gov. Whitmer"
and "Gretchen Whitmer" and "Whitmer" collapse to one canonical id); gazetteer geocoding to
county/district (dateline first, a curated notable-places override for newsworthy-but-small
locales, then a Census-Gazetteer-derived place list); and a hash-chain provenance stamp
computed before insert.

Nothing in extracted_facts or polls is ever deleted. That table is the permanent record
the 1950-archive and the time-capsule scrubber both read from.

§ The forecast, and its gate
Forecasting sits on top of the averages and fundamentals, itself deterministic — but a race-
type only earns visible forecasts after a nightly backtest, replayed against the archive back
to 1950, clears a Brier-score ceiling over a minimum number of graded past predictions. That
backtest is pure SQL and arithmetic, deliberately not an LLM: no peeking at outcomes that
haven't happened yet in the replay, no hindsight. The scorecard shows every race-type's
track record, including the ones still failing — the same "the honesty is the moat" principle a
serious forecasting product has to earn, not assert.


§ Race narrative generation
One LLM call per race-cluster update: a strict JSON contract — what_changed ,
why_it_might_have_changed (grounded in cited facts, never invented), what_to_watch ,
confidence . Malformed output gets exactly one retry, then a deterministic templated
summary built straight from the numbers. A devil's-advocate pass may only lower stated
confidence, enforced by a rank comparison in code — never by trusting the model's own
claim.

The genius layer: quantitative × qualitative
How PollGrid gets smarter than a poll average without ever letting the model touch a
number.



Every existing aggregator leaves the same thing on the table. RealClearPolitics-style sites
average and stop. 538-style forecasters blend polls with fundamentals — genuinely
quantitative, but structurally blind to everything that hasn't shown up in a poll yet: a scandal
breaking this morning, a coalition visibly cracking in coverage, a challenger's debate
performance landing badly. 270ToWin has no model at all. The obvious fix — "ask an LLM
who's going to win" — is exactly what this project's own hard rule forbids, and for good
reason: it would produce an unaccountable black box no more trustworthy than a pundit's
gut, just faster.

The actual fix is a pipeline, not a shortcut.


§ Step one: the Qualitative Factor Scorecard
Every race carries a fixed, named vector of scores — one per factor in the factors taxonomy
— re-scored by the LLM only when new relevant facts land for that race, cached otherwise.
Every score is numeric or ordinal, every score is cited back to the specific ingested fact(s)
that justified it, and the scoring runs against a fixed rubric per factor (never open-ended
"what do you think happens here"). If no LLM is available, every LLM-scored factor defaults
to a neutral 0 — the scorecard degrades to "quantitative-only," never blocks, never guesses.
This is a strict generalization of the causal-narrative pattern already established elsewhere in
this design: same one-call-per-update discipline, same cited-and-cached rule, same
deterministic neutral fallback.


§ Step two: a real, named, auditable combination
The Qualitative Factor Scorecard becomes plain numeric features — nothing more — fed
into a penalized logistic regression (elastic-net) stacking model alongside the existing
deterministic quantitative layer: the poll-average-plus-fundamentals blend's own probability,
and the demographic coalition detector's output. This is a real, well-understood statistical-
learning technique (a meta-learner over interpretable base predictions, the same family of
idea behind any serious ensemble forecast) — chosen specifically because elastic-net
regularization shrinks a factor's fitted weight toward zero if it isn't actually earning its keep,
rather than a human having to hand-prune a factor that "should" matter but doesn't. The
model itself is refit against the historical archive using the same replay-based
backtest machinery that already gates the quantitative-only forecast — no LLM
anywhere near the fitting process, no hindsight, coefficients stored as plain data and fully
inspectable.


§ Step three: it has to earn the right to be shown — against its own baseline

   THE RULE THAT KEEPS THIS HONEST


   A qualitative-augmented forecast for a given race-type category is only ever shown live if it beats the quantitative-only baseline's
   Brier score in the same nightly backtest that already gates every forecast category. The platform is not allowed to prefer the more
   impressive-sounding model — only the one that's demonstrably more calibrated, category by category. If the qualitative layer doesn't
   earn its keep for, say, gubernatorial races, gubernatorial forecasts simply run quantitative-only until it does. As more graded Factor-
   Scorecard-to-outcome pairs accumulate in the historical archive, the elastic-net weights refit periodically and the model can genuinely
   get more accurate over time — real statistical learning from a growing track record, not a vague claim that "the AI learns."

§ Why this is a structural advantage, not a marketing claim
No existing aggregator does this end to end. 538-style models are quantitative-only by
construction — qualitative judgment enters, if at all, as an analyst's ad hoc adjustment, not a
systematized, cited, backtested pipeline running across every tracked race. RCP-style sites
don't model at all. 270ToWin has no forecast, only a canvas for the user's own assumptions.
The novel mechanism here isn't "an LLM helps predict elections" — it's that the LLM's only
job is turning an ocean of unstructured text into structured, cited, rubric-scored data at a
scale no team of human analysts could sustain across hundreds of races simultaneously,
and a transparent, refittable, backtested statistical model does the actual predicting on top of
it.

The Analyst
Ollama-powered, fed automatically by every demographic and historical fact the
platform holds — reasoning, not inventing.



This is the "predictive genius" layer, and it's built the way GlobeGrid's grounded Q&A analyst
was built: the model never gets to make a number up, because it never has to — it's handed
a fully-assembled, cited context bundle before it writes a word. Ask it "why is this district
tightening" or "what does the demographic trend in this county actually predict" and it
reasons over real data, not vibes.

Ollama, local, by default. The same model already running on your machine drives race
narratives, dossier synthesis, and every Analyst answer, with cloud providers wired in only as
an optional fallback if Ollama isn't reachable. Nothing about the "genius" experience requires
an API key or a monthly bill.


§ How the context gets assembled — automatically
Every sync job that lands new Census, FEC, Congress.gov, or certified-results data doesn't
just update a table — it invalidates the standing context pack for every geography and race
it touches, but does not eagerly rebuild it. A context pack is a deterministically-built bundle:
the full demographic profile at every tier from precinct up through state, the historical result
and turnout trend where it exists, the current poll average, fundamentals snapshot, and
Qualitative Factor Scorecard (The genius layer), the fundraising trend, and any tagged
rhetoric/news facts — everything the Analyst is allowed to reason over, nothing it has to
guess. No LLM call happens until this bundle exists; if a tier's data is thin, the bundle says so
honestly instead of the model quietly filling the gap.


§ Sized for a real machine, not a hypothetical one
Default model: Qwen3-30B-A3B-Instruct — a mixture-of-experts model with 30.5B total
parameters but only 3.3B active per token, meaning it runs at roughly 8B-class speed on
ordinary hardware despite its 262K-token context window, which is exactly the "large context,
still fast locally" combination this feature needs. Constrained-hardware fallback: Qwen3
14B at 128K context, for machines without much RAM to spare. Optional heavy upgrade:
Llama 3.3 70B, for anyone running a Mac Studio or a dual-GPU rig who wants the strongest
reasoning available locally. All three are real, current Ollama-pullable models — this isn't
aspirational.

A context pack targets roughly 28,000–32,000 tokens — nowhere near any of the three
models' actual context ceilings, deliberately, because stuffing a window to its limit adds
latency and measurably dilutes attention quality even on long-context models. Budgeted
roughly as: demographics across the relevant tiers plus comparison baselines (~4,000
tokens), historical trend (~4,000), current polling/fundamentals/forecast snapshot (~3,000),
the Qualitative Factor Scorecard (~2,000), tagged news/rhetoric facts (~10,000 — the largest
single allocation, and the one most likely to overflow), leaving headroom for conversation
history and generation. When tagged facts alone would exceed their budget, the named
policy is relevance-ranked truncation: facts are pre-scored by the same embedding-
similarity mechanism the correlation engine already runs, the highest-relevance facts within
budget are kept, and recency is the tiebreak only among facts of similar relevance — never a
blind first-N or last-N cutoff.

Concurrency: Ollama serves one generation at a time per loaded model, so the real design
problem is queueing, not parallelism. A two-lane priority queue solves it: interactive requests

The LLM boundary
Stated once here, then enforced file by file everywhere it's used.



   ALLOWED


   Race-narrative prose (grounded, cited, never a call) · candidate/party dossier synthesis (background, cached, always over a curated

   Poll averaging · the fundamentals model · ideology scoring · forecasting math · the elastic-net weights that combine the Factor Scorecard

   calling decision itself.

Articles, everywhere
The news feeder doesn't wait for coverage to arrive — it goes and finds it, race by race,
candidate by candidate.



A generic RSS firehose (Ingestion layer) will always miss the long tail — down-ballot House
coverage, local-outlet reporting on a competitive statewide race, anything no national wire
happens to pick up. Nobody curates a feed for "every article about NC-13." So PollGrid
doesn't rely on one: it actively hunts.


§ The race search profile
Every tracked race carries a search profile, built the moment the race is seeded and kept
current automatically: race-name variants (the official name, common shorthand, "{state}
{office} race," "{state} {office} election {cycle}"), election-cycle tags, and every candidate's
name variants — full legal name, common nickname, title-plus-surname ("Sen. Ossoff"), and
alias forms pulled straight from FEC filing name variants. The moment a new candidate's
official filing record lands — FEC or state SoS — their names are added to the profile
automatically. A candidate gets searched for from the instant they're a real filer, not once
they're already famous enough for a generic feed to stumble onto.


§ Retrieval — deliberately not GDELT
GDELT is excluded on purpose. GlobeGrid's own history is the reason why — it was retired
after reportedly "trashing the system" with low-quality volume, and that lesson carries straight
over. The default, free, keyless mechanism is per-query Google News RSS
( news.google.com/rss/search?q={query} ) run on the same interval discipline as any other
source; a real commercial News API is wired in as an optional paid upgrade, mirroring the
exact same free-first/paid-upgrade shape already used for Ollama-vs-cloud and native-
feeds-vs-AP.


§ Dedup, relevance, and not becoming spam
Targeted-search hits land in the same raw_items table under the same
UNIQUE(source_id, external_id) dedup as the generic firehose, so an article both paths
find isn't double-counted — it just carries a matched_profile_id marker recording which
race/candidate profile(s) matched, for traceability (one article can legitimately match several,
e.g. two candidates in the same debate story). Nothing is filtered pre-ingestion on a
relevance guess; instead, every hit flows through the identical extraction/correlation pipeline
as everything else, and a genuinely irrelevant hit simply fails to correlate strongly and fades
out of relevance-ranked views on its own. A profile whose query returns nothing for a stretch
is logged and health-tracked exactly like a quiet source elsewhere in this design — never
retried in a tight loop.


§ On every race and candidate page
An Articles panel — the same "everywhere" pattern as polls — sits on every race, candidate,
and party page, pre-filtered to that entity, sorted by recency or relevance, each row showing
the outlet, its reliability tier, and a direct link, never a paraphrase. Route:
/api/articles/{entity_type}/{id} .

§ Real DDL for the geography tier — the primary-key question, settled
Recommendation: use the Census Bureau's own GEOID/FIPS codes as primary keys
wherever one already canonically exists (states, counties, congressional districts) rather
than inventing a surrogate integer — it makes every join against Census/TIGER data direct
with no separate mapping layer to maintain. Precincts get a surrogate INTEGER PRIMARY
KEY instead, because no federal agency defines a canonical precinct ID; a state's own
precinct code, where one exists, is kept as a plain column.




                                                                      34 / 40
                                                     PollGrid — Clean-Slate Architecture




CREATE TABLE states (
  fips_code        TEXT PRIMARY KEY,            -- 2-digit Census FIPS, e.g. '06'
  usps_code        TEXT NOT NULL UNIQUE,        -- 'CA'
  name             TEXT NOT NULL,
  elector_method   TEXT NOT NULL CHECK (elector_method IN ('winner_take_all','congressional_district')),
  electoral_votes INTEGER NOT NULL,
  is_territory     INTEGER NOT NULL DEFAULT 0
);


CREATE TABLE congressional_districts (
  geoid            TEXT NOT NULL,               -- Census GEOID, e.g. '0601'
  congress_number INTEGER NOT NULL,             -- which Congress this boundary version was seated for
  state_fips       TEXT NOT NULL REFERENCES states(fips_code),
  district_number INTEGER NOT NULL,             -- 0 = at-large
  effective_from   TEXT NOT NULL,               -- ISO date
  effective_to     TEXT,                        -- NULL = current
  PRIMARY KEY (geoid, congress_number)
);


CREATE TABLE county_equivalents (
  geoid            TEXT PRIMARY KEY,            -- 5-digit Census FIPS, e.g. '06037'
  state_fips       TEXT NOT NULL REFERENCES states(fips_code),
  name             TEXT NOT NULL,
  type             TEXT NOT NULL CHECK (type IN ('county','parish','borough','census_area','independent_city'))
);


CREATE TABLE precincts (
  precinct_id            INTEGER PRIMARY KEY,
  county_geoid           TEXT NOT NULL REFERENCES county_equivalents(geoid),
  state_precinct_code TEXT,
  name                   TEXT,
  coverage_confidence TEXT NOT NULL CHECK (coverage_confidence IN ('measured','derived','thin_coverage')),
  source_tier            TEXT NOT NULL CHECK (source_tier IN ('native','vest','openelections','manual'))
);


CREATE TABLE demographics (
  id             INTEGER PRIMARY KEY,
  tier           TEXT NOT NULL CHECK (tier IN ('nation','state','congressional_district','county_equivalent','precinct')),
  entity_id      TEXT NOT NULL,                 -- polymorphic: the relevant tier's own PK
  as_of          TEXT NOT NULL,
  category       TEXT NOT NULL CHECK (category IN ('population_age','race_ethnicity','education','economic','housing_urbanicity'
  variable       TEXT NOT NULL,                 -- e.g. 'median_household_income'
  value          REAL,
  confidence     TEXT NOT NULL CHECK (confidence IN ('measured','derived')),
  source         TEXT NOT NULL,                 -- e.g. 'acs5_2024:B19013_001E'
  UNIQUE (tier, entity_id, as_of, category, variable)
);


CREATE TABLE political_history (
  id             INTEGER PRIMARY KEY,
  tier           TEXT NOT NULL CHECK (tier IN ('state','congressional_district','county_equivalent','precinct')),
  entity_id      TEXT NOT NULL,
  office         TEXT NOT NULL CHECK (office IN ('president','senate','governor','house')),
  cycle_year     INTEGER NOT NULL,
  winner_party TEXT,
  margin_pct     REAL,
  turnout_pct    REAL,
  confidence     TEXT NOT NULL CHECK (confidence IN ('measured','derived','uncertain')),
  source         TEXT NOT NULL,
  UNIQUE (tier, entity_id, office, cycle_year)
);
